# Gateway-Generated, Production-Pinned Client Profile Plan

Status date: 2026-07-15.

Status: **proposed; no implementation has started**.

Decision class: **large, cross-component, security-sensitive change**.

Owning components: `platform/services/gateway` owns profile policy, issuance,
generation, artifact signing, revocation, download authorization, and audit.
`platform/client` owns consumption of one compiled profile, strict TLS, local
credential use, and fail-closed startup. `platform/website` owns only operator
intent collection and presentation. `platform/shared` owns a versioned profile
schema only if it is needed by the generator and generated source.

## Executive decision

Replace the production client environment/file configuration loader with a
single typed profile generated by the gateway and compiled into each issued
client artifact. Production clients have no runtime configuration source: no
environment-variable override, configuration-file discovery, command-line
option, endpoint fallback, or runtime development-mode switch. Missing,
malformed, expired, or policy-inconsistent compiled data fails startup before a
network connection is attempted.

Development remains possible only as a separately compiled development
artifact. Its build identity and profile source are selected by the trusted
build job at compile time. Environment, flags, files, registry values, home
directories, and local markers must never select development behavior in a
production artifact. Production builds must not link a development loader.

This prevents ordinary runtime reconfiguration; it cannot prove that a locally
privileged user has not patched, replaced, debugged, or copied the executable,
or extracted a software-held credential. Embedded values are not secrets. The
gateway must still regard all client payloads as client-authored and untrusted.
The stronger product claim requires a separate endpoint-integrity program:
platform signing enforcement, hardware-backed non-exportable keys, measured
boot, and gateway-verified remote attestation. This plan must never claim that
compiled configuration alone prevents local binary tampering.

## Current baseline and gap

`platform/client/cmd/main.go` starts through `setupApp`.
`platform/client/cmd/setup.go` calls `platform/client/internal/config.Load`.
That loader reads `AGENT_*` variables and derives defaults. Its `Config` is
consumed by TLS setup and `agentquic.Client`.

| Current input | Existing use | Target owner/source |
| --- | --- | --- |
| environment/version | production and localhost validation; QUIC metadata | gateway-selected compiled build identity |
| QUIC endpoint and TLS server name | dial target and hostname verification | gateway-selected compiled profile |
| certificate/key/CA/command-key paths | local credential and key loading | compiled credential-reference policy; platform store |
| heartbeat, operation, handshake, idle, keepalive, reconnect | scheduling and QUIC supervision | compiled bounded transport policy |
| replay-ledger and auth-key paths | durable local replay protection | immutable local-security-state policy |
| legacy HTTP variables | startup rejection | remove with the external loader; never replace with fallback |

The repository has no current dashboard generation route, client artifact
builder, enrollment service, issuance database, or artifact verification.
This is therefore a new gateway-owned feature, not a client-only refactor. A
browser-only Generate button must not create a trusted local configuration file.

## Scope

### Goals

- Produce a signed client artifact bound to one approved profile and target.
- Compile endpoint, exact TLS DNS name, approved trust/key identifiers or
  public material, cadence, transport limits, replay-state policy, and build
  identity into production artifacts.
- Preserve TLS 1.3, explicit server-name validation, mTLS identity, signed
  audience-bound commands, replay protection, and QUIC-only transport.
- Keep authorization, issuance, audit, credential lifecycle, revocation, and
  policy enforcement at the gateway.
- Provide artifact checksum, SBOM, provenance, signature, expiry, and rotation
  evidence for every download.

### Non-goals

- Treating browser input, embedded profile fields, or mTLS alone as executable
  integrity proof or authorization evidence.
- Embedding CA private keys, gateway signing private keys, reusable fleet keys,
  or a client private key in source or an artifact.
- Allowing a production client to choose another endpoint, CA, TLS name,
  credential location, timeout profile, or development mode.
- Adding HTTP/WebSocket fallback, DNS alternate endpoints, localhost
  exceptions, insecure TLS, or an automatic recovery profile.
- Exposing generic builds from raw browser-provided compiler arguments, URLs,
  paths, environment maps, templates, tags, or source fragments.

## Required ADR gate

Before implementation, gateway-security, release, and operations owners must
approve an ADR that freezes:

| Decision | Required initial position |
| --- | --- |
| Profile format | Typed canonical versioned profile generated into package-private Go source or immutable linker data. No JSON/YAML/TOML file or runtime environment parser in production. |
| Binding | Canonical signed manifest includes schema version, issuance ID, target OS/architecture, build version, endpoint, TLS name, trust/key IDs, credential reference, policy version, issue/expiry, and artifact digest. |
| Credentials | Prefer per-device hardware-backed non-exportable keys. If a software key is unavoidable, issue a short-lived per-agent key via the platform credential store, document extraction risk, and support immediate revocation. |
| Trust rotation | Pin approved CA/public-key set and exact TLS DNS name. Rotation uses an explicit overlapping, versioned policy and expiry; never an environment override. |
| Signing | Build/release service signs artifact digest and manifest with a protected identity; platform installer/OS verifies normal code signatures. |
| Gateway acceptance | Gateway records certificate-to-issuance/policy bindings and rejects expired/revoked credentials, without mistaking this for binary-integrity proof. |
| Strong integrity | If required, specify supported hardware, measured claims, nonce freshness, verifier, privacy, revocation, and gateway failure enforcement for remote attestation. |
| Development | Separate build targets, trust roots, command keys, and signing identities. Production compilation rejects development inputs. |
| Lifecycle | Define enrollment, installation, renewal, rotation, endpoint migration, revocation, upgrade, rollback, loss recovery, and deletion. |

The ADR must update the present externalized-production-client-config claim in
the roadmap and operations documents only when this replacement exists.

## Threat model and invariants

| Data/actor | Classification | Authority |
| --- | --- | --- |
| Dashboard choices | operator-authored intent | request an artifact only after server authorization; never supply unchecked build inputs |
| Profile, issuance, manifest | server-authored | gateway validates policy, binds and audits issuance, revokes |
| Artifact/signature/checksum | server-authored release output | configure one build and prove provenance |
| Client mTLS certificate | authenticated credential | derive gateway agent identity; does not prove executable integrity |
| Client telemetry/results/files/screen/terminal data | client-authored | untrusted observations only |

Mandatory invariants:

1. Production binaries make no environment, flag, file, CWD, home-directory,
   registry, or command-line read for profile selection/override.
2. They dial exactly one compiled host-and-port with one explicit compiled DNS
   `ServerName`; production rejects IP literals and `localhost`.
3. TLS always requires TLS 1.3, a non-empty approved root set, one approved
   client credential, the compiled name, and `InsecureSkipVerify == false`.
4. Connection failures retry only the compiled endpoint under bounded compiled
   policy. They never consult another endpoint or transport.
5. Gateway identity derives solely from the verified peer certificate; profile
   claims cannot affect identity, authorization, event provenance, or command
   authority.
6. Development and production have separate cryptographic/operational roots;
   production cannot recognize a development profile as valid.
7. Local binary modification is not hidden by mTLS. Platform enforcement and,
   where required, remote attestation are distinct controls.

## Target architecture

```text
Browser (operator intent) -> authorized gateway issuance API -> policy/audit/store
                                                        |             |
                                                        v             v
                                           canonical profile      enrollment/key lifecycle
                                                        |
                                                        v
                                  isolated fixed-toolchain build worker
                                                        |
                                                        v
                             signed artifact + manifest + SBOM + checksum
                                                        |
                                                        v
                      authorized download/install -> compiled-profile client
                                                        |
                                                        v
                                      mTLS QUIC -> gateway trust boundary
```

The browser submits typed intent only. The gateway selects sensitive values and
passes an issuance ID or canonical server-authored profile to a least-privilege
ephemeral builder. The worker receives no arbitrary shell argument or template,
has fixed target allowlists and no master-key access, and returns only a signed
opaque artifact reference. The browser receives no secret material in UI,
URLs, logs, or analytics.

## Implementation phases

### 0. Design and contract

1. Publish the ADR, threat model, data classifications, lifecycle, error
   taxonomy, compatibility policy, and measurable acceptance criteria.
2. Inventory every client config read in `cmd/setup.go`, `internal/config`, and
   `internal/agentquic`; distinguish host-integration environment reads from
   configuration reads.
3. Define bounded typed profile fields, canonical serialization, profile digest,
   manifest shape, deterministic generation, target matrix, and allowlists.
4. Define production acceptance: all historical `AGENT_*` variables are unable
   to affect the effective profile; missing profile data fails before dial;
   runtime inputs cannot activate development; issuance records reconstruct who
   issued, downloaded, revoked, and replaced every artifact.

### 1. Gateway issuance and protected build

1. Add gateway-owned durable issuance records: immutable issuance/requester ID,
   approved policy, target, profile digest, certificate/public-key binding,
   artifact digest, signing key ID, timestamps, expiry, download state,
   revocation state, and audit events. Set retention/privacy limits.
2. Add authenticated and authorized create, inspect, download, revoke, and
   rotate routes. The dashboard currently lacks operator authentication; do not
   expose generation until that server-side boundary exists.
3. Implement an isolated fixed-toolchain builder: pinned dependencies,
   allowlisted targets, bounded resources/output/time, no arbitrary network
   egress, no gateway master-key access, and no shell construction from UI
   values.
4. Generate `profile_generated.go` (or immutable equivalent) in an ephemeral
   workspace. Never hand-edit generated source. Sign a canonical manifest after
   compilation and publish artifact checksum, SBOM, provenance, profile digest,
   signer/key ID, and expiry.
5. Integrate issuance revocation and certificate/key rotation. Endpoint
   migration produces a new artifact/profile, not a mutable fallback list.

### 2. Immutable production client

1. Replace `clientconfig.Load()` with a compiled production-profile provider.
   Refactor `Config` into an immutable profile with no production loader.
2. Compile endpoint, server name, public trust/verification material or IDs,
   cadence, QUIC limits, profile/issuance metadata, and build identity. Access
   private keys only through an immutable credential reference and platform
   secure-store adapter; never compile private keys or use an overrideable path.
3. Keep replay state only as authenticated security state at a location derived
   from compiled policy and secure OS storage, never an `AGENT_*` override.
4. Validate at startup and constructor boundaries: schema, target, mode,
   expiry, endpoint, DNS name, trust/key material, positive bounds, and timing
   relationships. Return non-secret failure categories before dialing.
5. Refactor TLS and `agentquic.New` to receive the profile and credential
   provider. Preserve strict equality checks for the compiled TLS server name.
6. Delete production `AGENT_*` reads, default certificate/state paths,
   `AGENT_ENVIRONMENT`, and legacy transport configuration parsing. Do not keep
   a compatibility parser, even one that ignores values.
7. Keep profile fields unexported, expose no setters, defensively copy bytes,
   and prohibit mutable package state or persisted replacement profiles.

### 3. Development isolation

1. Create distinct compile-time production and development profile packages or
   build targets. Production includes only generated production profile code;
   development override code is absent from the production link.
2. Reject development tags, fixtures, CAs, command keys, localhost names, and
   signing identities in the release builder. Use visibly distinct development
   artifact names/signing identities and do not publish them as releases.
3. Permit environment overrides only in development-only code, document them as
   unsafe test controls, and test their absence from production builds.
4. Add immutable mode metadata for audit only. Gateway policy rejects
   development credentials/profiles from production listeners.

### 4. Website workflow

1. Add Generate Client only after gateway authentication exists. Render only
   gateway-provided allowlisted choices (approved deployment policy and target),
   never raw endpoint, CA, certificate, path, environment, or timeout fields.
2. Use typed TanStack Query mutations with pending, success, partial, failure,
   duplicate prevention, accessible status, issuance ID, expiry, target,
   checksum, and rotation/revocation state.
3. Issue short-lived single-use audited download grants. Do not put credentials
   or secret material in browser responses; a download is not enrollment proof.

### 5. Rollout and retirement

1. Gate the issuance service internally until authorization, signing, audit,
   lifecycle, and platform tests pass.
2. Run staging issuance for every supported target; test clean install,
   reconnect, restart, rotation, migration, revocation, rollback, and incident
   recovery against the gateway issuance record.
3. Release new artifacts rather than mutating deployed binaries. Time-bound
   legacy externally configured clients and give generated clients no downgrade
   path.
4. Remove legacy loader code, examples, scripts, fixtures, and documentation
   only after migration evidence. Deployment validation fails if a production
   artifact links the legacy loader.

## Verification and evidence

### Tests

- Profile unit tests accept a complete approved profile and reject missing or
  unknown schema, empty/invalid endpoint, IP/localhost TLS name, empty trust,
  bad key ID, mode/target mismatch, expiry, invalid timings, and mutable alias.
- Client setup tests prove rejection occurs before credential load/network dial;
  TLS requires compiled server name, TLS 1.3, roots, and secure verification.
- Set every historical `AGENT_*` variable in regression tests and prove an
  unchanged production profile/dial target. Add a static CI check forbidding
  production profile imports of environment configuration APIs.
- Target-specific build tests prove development overrides work only in a
  development artifact and cannot be linked into production.
- Gateway tests cover authorization, allowlist rejection, canonical bytes and
  digest, audit/restart recovery, duplicate issuance, one-time download,
  expiry, revocation, rotation, and builder input isolation.
- Website tests cover typed bounded requests and all state-changing control
  states; disabled controls are not authorization evidence.

### Integration and release gates

- Rebuild a known issuance; verify compiled profile digest, manifest signature,
  checksum, SBOM/provenance, and target metadata against the gateway record.
- Run it in a clean environment polluted with legacy variables/files and
  alternate CWD/home directories. It must use only compiled policy or fail;
  it must never fall back.
- Attempt endpoint, CA, server-name, file, environment, flag, and local
  development-marker changes. None may change production behavior.
- Test untrusted signature/modified artifact rejection where OS signing
  enforcement exists, while documenting privileged-local bypass as residual risk
  unless remote attestation is implemented.
- Verify gateway rejection/auditing for revoked and expired credentials without
  trusting a client profile claim.
- Run `make test`, `make ci-go`, `make ci-web`, and `make ci`; run installation,
  signing, and credential-store tests on every supported OS.
- Complete threat-model, SBOM/dependency, reproducibility, artifact-signature,
  rollback, recovery, and manual incident-response evidence before production.

## Failure behavior

| Condition | Required behavior |
| --- | --- |
| Missing/corrupt/expired/invalid profile | Exit before dial; bounded non-secret error; no defaults/fallback. |
| Credential unavailable | Exit or stay disconnected under compiled bounded retry; never search paths or download replacement. |
| Endpoint unavailable | Retry only compiled endpoint; no alternate endpoint or transport. |
| Issuance/build/signing failure | Mark issuance failed and audit it; release no partial artifact. |
| Signature/provenance mismatch | Installer/release verifier rejects; revoke affected issuance on suspected compromise. |
| Key/profile compromise | Disable/revoke at gateway and issue a new artifact under incident procedures. |
| Local modification | Do not infer integrity from mTLS; use OS signing and remote attestation only when approved. |

## Delivery workflow

This is a large change. Branch `feature/generated-client-profile` from
`master`, make the first coherent ADR/profile-contract commit, push it, and
open a draft pull request. Reviewable slices:

1. `docs(security): define generated client profile contract`
2. `feat(gateway): add audited client profile issuance`
3. `build(client): generate signed profile-bound artifacts`
4. `refactor(client): consume compiled production profile`
5. `feat(website): add authorized client generation workflow`
6. `test(security): verify production profile immutability`
7. `docs(operations): document issuance rotation and recovery`

Every PR update must state component ownership, data/trust classification,
gateway enforcement, migration impact, residual local-tampering risk, and
validation evidence. Do not approve a stronger integrity claim until remote
attestation is independently designed and verified.

## Completion criteria

Completion requires a clean signed production artifact from an auditable
gateway issuance that uses exactly one compiled profile; no runtime input can
change its endpoint, TLS policy, credential reference, transport policy, or
mode; development code is absent; no fallback exists; gateway authorization,
issuance, lifecycle, and revocation are enforced; and all component, platform,
security, release, and operational evidence passes.
